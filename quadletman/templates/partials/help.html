{{ _("How quadletman works") }}

{{ _("A quick reference for managing containerised services on this host.") }}

{{ _("Compartments") }}

{{ _("Each compartment is an isolated Linux user account (") }}qm-{id}{{ _("). Containers, volumes, and network configuration inside a compartment are invisible to other compartments.") }}

{{ _("Systemd user units are managed per compartment.") }} loginctl linger {{ _("is enabled so services keep running after logout.") }}

{{ _("Unit files:") }} /home/qm-{id}/.config/containers/systemd/

{{ _("Containers") }}

{{ _("Containers are defined as Podman Quadlet") }} .container {{ _("unit files and managed by systemd. Each container is a systemd service — you can start, stop, restart, and enable autostart from the compartment view.") }}

{{ _("Images can be pulled from a registry, referenced from a pre-pulled") }} .image {{ _("unit, or built from a Containerfile stored in the compartment.") }}

{{ _("Volumes and mounts") }}

{{ _("Managed volumes") }}

{{ _("Created in the Volumes section of a compartment. quadletman allocates a host directory and applies the correct SELinux label automatically.") }}

/var/lib/quadletman/volumes/{compartment-id}/{volume-name}/

{{ _("Once created, a volume is mounted into a container from the Volumes tab of the container form. Each mount has an optional options field:") }}

U {{ _("— Podman remaps file ownership to the container's primary user. Use when the container runs as non-root and gets permission errors.") }}
idmap {{ _("— Same effect as U but using kernel ID-mapped mounts without modifying files on disk. Requires kernel 5.12+ and Podman 4.1+.") }}
ro {{ _("— Mount read-only.") }}
Z {{ _("— Apply a private SELinux label (only needed when SELinux is enforcing).") }}

{{ _("Options can be combined with commas, e.g.") }} U,ro.

{{ _("Quadlet-managed volumes") }}

{{ _("An optional mode for managed volumes. Instead of a plain host directory, quadletman generates a") }} .volume {{ _("unit file that Podman manages as a named volume. This enables advanced storage options: a custom driver, a specific device, and driver-level mount options.") }}

{{ _("Enable the Quadlet volume toggle when creating a volume to use this mode. Leave it off for the simpler host-directory mode, which is sufficient for most use cases.") }}

{{ _("Bind mounts") }}

{{ _("Mount any existing host path directly into a container — no volume needs to be created first. Configured in the Volumes tab of the container form.") }}

{{ _("Useful for exposing host configuration files, sockets, or device nodes. The same options apply as for managed volume mounts.") }}

{{ _("Certain sensitive paths are blocked:") }} /etc, /proc, /sys, {{ _("and other system directories cannot be bind-mounted.") }}

{{ _("Helper users") }}

{{ _("Rootless Podman maps container UIDs to host UIDs via a user namespace. By default, container UID 0 (root inside the container) maps to the compartment's Linux user") }} (qm-{id}), {{ _("and every other container UID maps to an unprivileged host UID drawn from that user's subUID range.") }}

{{ _("This works transparently for most containers, but causes a problem with volume ownership: if a container process runs as UID 1000, the files it writes to a volume are owned by the corresponding host subUID — a number like 165536 that no named user owns. Browsing or backing up those files from the host becomes awkward.") }}

{{ _("Helper users solve this. When you add a UID mapping for container UID N, quadletman creates a real Linux user") }} qm-{id}-N {{ _("whose host UID is exactly the subUID that Podman maps container UID N to. You can then set a volume's owner to that helper user, and the volume directory is owned by a named host account — no numeric UID hunting required.") }}

{{ _("Helper users are created and removed automatically by quadletman as you add or remove UID mappings. They appear in the Helper Users section of a compartment when at least one exists.") }}

{{ _("All users belonging to a compartment — the compartment root and every helper user — share a single Linux group named") }} qm-{id}. {{ _("Its GID is pre-filled in the volume form when creating a quadlet-managed volume. Setting the volume's group to this GID means any container UID that has a helper user can read and write the volume, which is useful when multiple containers in the same compartment need shared write access to the same directory.") }}

{{ _("Example — a container running as UID 1000:") }}
qm-myapp {{ _("— compartment user (UID), also the shared group name") }}
qm-myapp-1000 {{ _("— helper user for container UID 1000, member of qm-myapp group") }}

{{ _("Networking") }}

{{ _("Containers inside the same compartment share a network and can reach each other on") }} localhost{{ _(". Containers in different compartments are isolated from each other — each compartment runs as a separate Linux user with its own rootless Podman network.") }}

{{ _("Cross-compartment communication") }}

{{ _("Option 1 — Publish a port (recommended)") }}

{{ _("Publish a port from the providing container:") }}

PublishPort=8080:8080

{{ _("Then connect from the other compartment using the hostname Podman provides inside every container:") }}

host.containers.internal:8080

{{ _("This resolves to the host's gateway address as seen from inside the container, reaching any published port on the host. No extra setup required.") }}

{{ _("Option 2 — Root-created shared Podman network") }}

{{ _("A network created as root can in principle be shared across rootless users. Each compartment's containers would reference the same named network:") }}

Network=shared-net

{{ _("This requires creating and configuring the network on the host outside of quadletman. A root-level compartment is not available through this interface — the setup must be done manually on the host.") }}
